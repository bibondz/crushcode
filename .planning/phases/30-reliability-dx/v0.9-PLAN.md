## Phase 31: Hierarchical Config

### Problem
Single config file at `~/.crushcode/config.toml`. No project-local overrides.

### Current State
- `Config` struct has ~15 fields (provider, model, api_keys, mcp_servers, etc.)
- `loadOrCreateConfig()` loads from single path
- `getConfigDir()` checks XDG_CONFIG_HOME then ~/.config/crushcode
- `project.zig` already has `.crushcode/` directory support (for instructions.md)
- `detectProject()` identifies project types but doesn't load config

### Implementation
1. Add `loadProjectConfig()` to `project.zig` — reads `.crushcode/config.toml` from CWD
2. Add `Config.merge(override: Config)` — shallow merge where non-default override fields replace base
3. Modify `loadOrCreateConfig()` to: load user config → load project config → merge → apply env vars
4. Add `CRUSHCODE_CONFIG` env var support (already exists in `getConfigPath()` but needs verification)
5. Priority: `.crushcode/config.toml` (project) > `~/.crushcode/config.toml` (user) > env vars (runtime)

### Merge Rules
- Strings: override wins if non-empty
- Numbers: override wins if non-zero (except temperature=0 is valid — use optional)
- HashMaps: merge (project keys override user keys)
- Arrays: project replaces user (MCP servers are project-specific)

### Key Files
- `src/config/config.zig` — Config struct, loadOrCreateConfig(), add merge()
- `src/config/env.zig` — getConfigDir()
- `src/config/project.zig` — add loadProjectConfig()

## Phase 32: File Tracker

### Problem
AI re-reads files it already knows, wasting tokens. No caching or change detection.

### Current State
- `FileReader` reads fresh every time, no caching
- `SourceTracker` is for knowledge provenance, not file caching
- `executeReadFileTool` has no tracking — always reads from disk

### Implementation
1. Create `src/core/file_tracker.zig`:
   ```zig
   pub const TrackedFile = struct {
       path: []const u8,
       hash: u64,           // FNV-1a content hash
       size: u64,
       mtime: i128,         // modification time
       read_count: u32,
       last_read: i64,      // timestamp
       content_snapshot: ?[]const u8 = null,  // optional cache
   };
   pub const FileTracker = struct {
       files: std.StringHashMap(TrackedFile),
       max_cache_bytes: usize = 512 * 1024,  // 512KB cache limit
       // Methods: track(), isUnchanged(), getCached(), invalidate(), deinit()
   };
   ```
2. Integrate with `executeReadFileTool` in tool_executors.zig:
   - Before reading: check `tracker.isUnchanged(path)` — if hash matches, return cached content
   - After reading: `tracker.track(path, content, stat)`
3. Threadlocal `active_file_tracker` pattern (same as shell_state, blocklist, etc.)
4. Auto-invalidate on file edit (edit tool calls `tracker.invalidate(path)`)

### Key Files
- NEW `src/core/file_tracker.zig` — FileTracker struct
- `src/chat/tool_executors.zig` — integrate in executeReadFileTool, invalidate on write/edit

## Phase 33: Ripgrep Fallback

### Problem
`grep -rn` is slow on large codebases. Ripgrep is 10-100x faster.

### Current State
- `executeGrepTool` uses `grep -rn --binary-files=without-match` via shell
- Shell execution pattern via `std.process.Child` well-established
- Supports: pattern, path, include, max_results

### Implementation
1. Modify `executeGrepTool` to try `rg` first:
   ```zig
   // Try ripgrep first
   const rg_result = tryExecuteRg(allocator, pattern, path, include, max_results);
   if (rg_result) |result| return result;
   
   // Fallback to grep
   return tryExecuteGrep(allocator, pattern, path, include, max_results);
   ```
2. `tryExecuteRg()`: runs `rg --no-heading --line-number --color=never {include_flag} --max-count={N} {pattern} {path}`
   - Use `std.process.Child` directly (not through `sh -c`) for safety
   - If `rg` not found (exit code 127 or spawn error), fall through to grep
3. `include` flag mapping: `--include=*.zig` → `--glob=*.zig` for rg
4. Result format stays identical to current grep output

### Key Files
- `src/chat/tool_executors.zig` — modify executeGrepTool()

## Phase 34: Log CLI

### Problem
No file-based logging. No way to inspect past sessions or debug issues after the fact.

### Current State
- `std.log` level `.warn` in main.zig, all output to stderr
- 148+ std.log calls across 29 files
- No log files, no structured format, no rotation

### Implementation
1. Create `src/core/structured_log.zig`:
   ```zig
   pub const StructuredLogger = struct {
       log_dir: []const u8,
       current_file: ?std.fs.File,
       current_size: usize,
       max_file_size: usize = 10 * 1024 * 1024,  // 10MB
       max_files: u32 = 5,  // rotation
       // Methods: init(), log(), rotate(), deinit()
       // Format: JSONL: {"ts":..., "level":"info", "msg":"...", "fields":{...}}
   };
   ```
2. Override Zig's `std.log` via `pub fn logFn()` in main.zig to write to both stderr AND structured log file
3. Log file location: `~/.crushcode/logs/crushcode-{date}.jsonl`
4. Create `src/commands/logs.zig`:
   - `crushcode logs` — show last 20 entries
   - `crushcode logs --tail N` — show last N entries
   - `crushcode logs --follow` — tail -f equivalent (poll file for new lines)
   - `crushcode logs --level error` — filter by level
   - `crushcode logs --session <id>` — filter by session ID field
5. Register in CLI registry as `logs` command

### Key Files
- NEW `src/core/structured_log.zig` — JSONL logger with rotation
- NEW `src/commands/logs.zig` — log CLI command
- `src/main.zig` — override logFn
- `src/cli/registry.zig` — register logs command

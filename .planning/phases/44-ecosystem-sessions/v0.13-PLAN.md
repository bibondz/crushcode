# v0.13.0 — Ecosystem + Sessions

**Milestone:** v0.13.0
**Focus:** MCP Server Exposition + Session Management CLI
**Date:** 2026-04-17

---

## Goal

Make crushcode a first-class citizen in the AI tooling ecosystem by exposing its capabilities as an MCP server, and give users full session management visibility via CLI commands.

---

## Phase 44: Session Management CLI

**Effort:** Low — all infrastructure exists, just needs CLI exposure

### What exists
- `src/session.zig`: `saveSession()`, `loadSession()`, `deleteSession()`, `listSessions()`, `generateSessionId()`, `isInterrupted()`
- `src/cli/args.zig`: `--continue` and `--session <id>` flags
- Sessions stored as JSON in `~/.crushcode/sessions/`
- Session struct: id, title, messages, model, provider, total_tokens, total_cost, turn_count, duration_seconds

### What to build
1. **Create `src/commands/session_cmd.zig`** — Session command handler
   - `crushcode sessions list` — List all sessions with table output (ID, title, model, turns, tokens, cost, updated)
   - `crushcode sessions show <id>` — Display full session metadata (messages summary, timing, cost breakdown)
   - `crushcode sessions delete <id>` — Delete a session file (uses existing `deleteSession()`)
   - `crushcode sessions clean [--days N]` — Remove sessions older than N days (default 30), remove interrupted sessions
   - `crushcode sessions` (no subcommand) — Default to `list`

2. **Register in `src/cli/registry.zig`** — Add `"sessions"` command

3. **Wire in `src/commands/handlers.zig`** — Add `handleSessions` dispatcher

4. **Update `src/main.zig`** — Add sessions to help text

### Files to create/modify
- **CREATE** `src/commands/session_cmd.zig`
- **MODIFY** `src/cli/registry.zig` — add sessions command
- **MODIFY** `src/commands/handlers.zig` — add handleSessions dispatch
- **MODIFY** `build.zig` — add session_cmd module

### UAT criteria
- `crushcode sessions list` shows table of sessions sorted by updated_at desc
- `crushcode sessions show <id-prefix>` works with partial ID match
- `crushcode sessions delete <id>` removes the file
- `crushcode sessions clean` removes sessions older than 30 days
- `crushcode sessions` defaults to list

## Phase 45: MCP Server Core

**Effort:** High — new subsystem, but can reuse transport patterns from `src/mcp/transport.zig`

### What exists
- `src/mcp/client.zig` — Full MCP client (connects TO servers)
- `src/mcp/transport.zig` — Transport types (stdio, SSE, HTTP), JSON-RPC message helpers, `cloneJsonValue`, `deinitJsonValue`
- `src/mcp/bridge.zig` — Bridge between crushcode and external MCP servers
- `src/tools/registry.zig` — `ToolRegistry` with tool definitions, categories, enable/disable
- `src/chat/tool_executors.zig` — 6 tool executors (bash, file_edit, file_write, file_read, glob, grep)
- `src/protocol/tool_types.zig` — Tool type definitions

### What to build
1. **Create `src/mcp/server.zig`** — MCP server implementation
   - `MCPServer` struct with JSON-RPC 2.0 request handler
   - Handle `initialize` method — return server capabilities (tools)
   - Handle `tools/list` method — return crushcode's built-in tools as MCP tool schemas
   - Handle `tools/call` method — dispatch to appropriate tool executor
   - Handle `ping` method — health check
   - Error responses per JSON-RPC 2.0 spec (-32600 invalid, -32601 method not found, -32602 invalid params)

2. **Create `src/mcp/tool_exposition.zig`** — Tool schema generation
   - Convert crushcode's internal `Tool` structs to MCP tool schemas (name, description, inputSchema)
   - Map tool categories to MCP tool types
   - Support for: bash (shell execution), file_read, file_edit, file_write, glob, grep
   - Each tool gets JSON Schema for input parameters

3. **Integration with existing systems**
   - Reuse `ToolRegistry` from `src/tools/registry.zig` for tool listing
   - Reuse tool executor functions from `src/chat/tool_executors.zig` where possible
   - Respect permission modes — server mode should use a configurable permission level

### Files to create
- **CREATE** `src/mcp/server.zig` — MCPServer with JSON-RPC request routing
- **CREATE** `src/mcp/tool_exposition.zig` — Tool-to-MCP-schema conversion

### UAT criteria
- MCPServer can parse JSON-RPC 2.0 requests and route to handlers
- `initialize` returns proper capabilities response
- `tools/list` returns crushcode tools as MCP schemas
- `tools/call` can execute bash commands and return results
- Invalid requests get proper JSON-RPC error responses
- Unit tests for all JSON-RPC handlers

## Phase 46: MCP Server Transport + CLI Integration

**Effort:** Medium — stdio transport is straightforward, HTTP transport builds on existing http_client patterns

### What to build
1. **Add to `src/mcp/server.zig`** — Transport listeners
   - **stdio transport**: Read JSON-RPC from stdin, write responses to stdout. This is the primary MCP transport.
   - **HTTP transport**: POST endpoint accepting JSON-RPC requests (reuse patterns from transport.zig)
   - Server lifecycle: start, accept connections, handle requests, graceful shutdown

2. **CLI integration**
   - `crushcode mcp serve [--transport stdio|http] [--port 8080]` — Start MCP server
   - Default to stdio transport (most MCP clients expect this)
   - HTTP transport for remote access
   - Config file support: `~/.crushcode/mcp_server.toml` for server configuration

3. **Update existing files**
   - Add `serve` subcommand to `src/commands/mcp_handler.zig`
   - Add server module to `build.zig`
   - Update help text

4. **Documentation**
   - Add `--help` output showing how to use crushcode as MCP server
   - Example: `crushcode mcp serve --transport stdio` for IDE integration

### Files to create/modify
- **MODIFY** `src/mcp/server.zig` — add transport listeners
- **MODIFY** `src/commands/mcp_handler.zig` — add `serve` subcommand
- **MODIFY** `build.zig` — wire server and tool_exposition modules
- **MODIFY** `src/commands/handlers.zig` — update help text

### UAT criteria
- `crushcode mcp serve` starts stdio-based MCP server
- External MCP client can connect via stdio and call tools
- `crushcode mcp serve --transport http --port 8080` starts HTTP server
- `curl localhost:8080` with JSON-RPC request returns proper response
- Server handles multiple sequential requests
- Graceful shutdown on SIGINT/SIGTERM

## Implementation Order

```
Phase 44 (Session CLI) → Phase 45 (MCP Server Core) → Phase 46 (Transport + CLI)
```

Phase 44 is independent and quick. Phase 45 builds the core server logic. Phase 46 adds transport and CLI integration.

Phase 44 can be delegated to a `quick` agent.
Phases 45-46 should each go to `deep` agents — they're substantial new subsystems.

## Risk Assessment

| Risk | Mitigation |
|------|------------|
| Tool executor reuse — chat tool executors may be tightly coupled to chat session state | MCP server may need simplified wrappers around tool logic |
| stdio transport blocking — reading from stdin is blocking in Zig | Use buffered reader with line-based parsing |
| Permission model in server mode — who approves tool calls? | Default to a configurable permission mode; server mode uses `--permission` flag |
| JSON-RPC batch requests — MCP spec supports them | Start with single-request handling, add batch support later |

## Phases

### Phase 47: Graph Algorithms (PageRank, Clustering, Bridge Detection)
**Source**: Firstbrain — `graph-engine.cjs`
**Depends on**: Nothing (builds on existing `src/graph/`)
**Priority**: High

**Goal**: Implement core graph algorithms in native Zig that enable crushcode to understand codebase structure — identify important files, find hidden connections, detect architectural hubs.

**Scope**:
- Add `src/graph/algorithms.zig` with:
  - PageRank (damping 0.85, configurable iterations)
  - Jaccard similarity for structural equivalence
  - Bridge detection for critical connecting nodes
  - Multi-hop connection discovery (2-3 hops)
  - Topic clustering via tag co-occurrence
- Extend existing `src/graph/graph.zig` KnowledgeGraph with:
  - `computePageRank()` → ranks nodes by importance
  - `findBridges()` → identifies critical connecting nodes
  - `clusterByTags()` → groups related nodes
  - `findSimilar(node_id)` → structural similarity search
  - `findPaths(from, to, max_hops)` → multi-hop connections
- Add `crushcode graph analyze` CLI command
- Add `crushcode graph important` (top-N by PageRank)
- Add `crushcode graph bridges` (critical nodes)
- Unit tests for each algorithm

**Out of scope**: Visual graph rendering, graph persistence to disk

### Phase 48: Knowledge Operations (Ingest/Query/Lint)
**Source**: karpathy-second-brain — wiki schema, Firstbrain — memory architecture
**Depends on**: Phase 47 (graph algorithms enhance query)
**Priority**: High

**Goal**: Implement knowledge lifecycle operations — ingest raw knowledge, query with citations, lint for quality.

**Scope**:
- Create `src/knowledge/` module:
  - `schema.zig` — KnowledgeNode type with YAML frontmatter, tags, sources, timestamps
  - `ingest.zig` — Process source files into structured knowledge nodes
  - `query.zig` — Search + synthesize knowledge with citation tracking
  - `lint.zig` — Health checks (broken links, orphans, contradictions, stale data)
  - `vault.zig` — Vault management (raw/ + wiki/ separation, index.md generation)
- Add CLI commands:
  - `crushcode knowledge ingest <path>` — Process source into knowledge base
  - `crushcode knowledge query <question>` — Search knowledge base
  - `crushcode knowledge lint` — Health check knowledge base
  - `crushcode knowledge status` — Show vault statistics
- Wire into existing `src/graph/` for relationship tracking
- Unit tests for each operation

**Out of scope**: LLM-powered synthesis (that's Phase 49), background ingestion

### Phase 49: Worker Agent Architecture + Multi-Model Routing
**Source**: COG-second-brain — worker agents, DeepTutor — two-layer plugin model
**Depends on**: Nothing (independent of graph/knowledge)
**Priority**: High

**Goal**: Enable crushcode to spawn specialized worker agents that execute tasks in parallel, with intelligent model routing.

**Scope**:
- Create `src/agent/worker.zig`:
  - `WorkerAgent` struct with specialty, model preference, output path
  - `WorkerPool` for managing concurrent workers
  - File-based communication pattern (worker writes to /tmp/, returns status + path)
  - Maximum concurrent workers (configurable, default 5)
  - Worker timeout and cleanup
- Create `src/agent/router.zig`:
  - `ModelRouter` with task-type → model mapping
  - Rules: data-heavy tasks → fast model, reasoning → premium model
  - Cost-aware routing (prefer cheaper models when quality difference is small)
  - Integration with existing `ProviderRegistry`
- Create `src/agent/capability.zig`:
  - `Capability` struct (Level 2 — multi-step agent pipeline)
  - `Tool` integration with existing tool registry (Level 1)
  - `UnifiedContext` flowing through all operations
- Add CLI commands:
  - `crushcode agent spawn <type> <task>` — Spawn a worker agent
  - `crushcode agent list` — List active workers
  - `crushcode agent status <id>` — Check worker status
- Unit tests

**Out of scope**: Full multi-agent orchestration DSL, inter-agent messaging

### Phase 50: Context Optimization + Governance Zones
**Source**: claude-token-efficient — optimization rules, Firstbrain — governance system, OpenHarness — permissions
**Depends on**: Nothing (independent)
**Priority**: Medium

**Goal**: Reduce token usage through prompt optimization rules and add three-zone governance for safe autonomous operation.

**Scope**:
- Create `src/ai/context_optimizer.zig`:
  - `ContextOptimizer` struct with rule-based compression
  - Rules: remove sycophancy, redundant explanations, restatements
  - Profile system: coding, analysis, compact modes
  - Integration with existing `ContextBudget` system
  - Token savings estimation before/after optimization
- Create `src/permission/governance.zig`:
  - `GovernanceZone` enum: AUTO, PROPOSE, NEVER
  - `GovernancePolicy` mapping operations to zones
  - Default policies:
    - AUTO: read files, lint fixes, update memory, search
    - PROPOSE: code edits, new files, config changes
    - NEVER: delete files, force push, merge to main
  - Integration with existing `PermissionSystem`
- Create `src/permission/sensitive_paths.zig`:
  - Built-in protection for SSH keys, AWS/GCP credentials, .env files
  - Glob-pattern path rules
  - Command deny patterns
- Add CLI commands:
  - `crushcode governance list` — Show zone policies
  - `crushcode governance check <operation>` — Check which zone applies
- Add `/compact` slash command enhancement with optimization profile selection
- Unit tests

**Out of scope**: Learning-based optimization, adaptive governance

## Risk Mitigation

- **Graph algorithms in Zig**: No external dependencies needed — pure math implementations
- **Worker agents**: Start simple (subprocess-based), don't over-engineer IPC
- **Context optimization**: Start with rule-based (deterministic), don't attempt ML-based
- **TUI widgets**: libvaxis is already a dependency — just need to integrate, not build new
